# APK Compat Helper UI 设计说明

## 1. 设计目标

设计一个面向普通用户的跨平台桌面小工具. 用户完成核心任务时只执行以下流程:

```text
选择 APK -> 查看自动分析结果 -> 点击开始修复 -> 获取新 APK
```

保持界面简单, 可信, 安静. 不使用仪表盘, 侧边栏, 多页面导航或大量技术详情.

## 2. 产品定位

- 产品名称: APK Compat Helper.
- 产品类型: macOS 和 Windows 本地桌面工具.
- 核心功能: 分析旧 Android APK, 调整兼容参数, 重建并签名新 APK.
- 目标用户: 不熟悉 Android开发的小白用户和需要快速处理旧 APK的普通用户.
- 使用特点: 单次只处理一个 APK, 操作频率低, 参数通常使用默认值.

## 3. 设计原则

- 将默认路径压缩为一次文件选择和一次主按钮点击.
- 默认隐藏 targetSDK和规则选项.
- 使用用户能理解的结果, 不直接堆叠 Manifest, DEX或系统行为术语.
- 明确告诉用户原 APK不会被覆盖.
- 明确告诉用户输出 APK会重新签名.
- 将风险作为提示, 不因普通兼容风险阻止用户继续.
- 只在 APK损坏, 输入不完整或存在安全边界时禁止修复.

## 4. 视觉方向

### 4.1 风格

- 使用轻量桌面工具风格.
- 使用浅灰背景和白色主面板.
- 使用单个橙色主操作色.
- 使用绿色表达解析完成和可修复状态.
- 使用细边框, 轻阴影和中等圆角.
- 避免玻璃拟态, 渐变文字, 霓虹效果和复杂动画.
- 避免将界面设计成开发者控制台或安全扫描仪.

### 4.2 颜色

| Token | 颜色 | 用途 |
|---|---|---|
| `background` | `#EDF1F2` | 应用背景 |
| `surface` | `#FFFFFF` | 主面板和输入区域 |
| `text-primary` | `#20272B` | 标题和正文 |
| `text-secondary` | `#6D797F` | 辅助信息 |
| `border` | `#D6DDDF` | 分隔线和控件边框 |
| `success` | `#24735A` | 已解析, 可修复和推荐状态 |
| `success-surface` | `#E8F2EE` | 成功提示背景 |
| `primary-action` | `#DC6239` | 主修复按钮 |
| `warning` | `#A85A2C` | 安全敏感参数提示 |

不要增加第二个高饱和强调色.

### 4.3 字体

使用系统字体, 不依赖在线字体:

```css
font-family: "Avenir Next", "Segoe UI Variable", "PingFang SC", "Microsoft YaHei", sans-serif;
```

字号建议:

- 页面标题: 27 到 38 px, 700.
- 模块标题: 13 到 14 px, 600或700.
- 正文: 11 到 13 px.
- 辅助文字: 9 到 10 px.
- 按钮文字: 14 px, 700.

## 5. 窗口与布局

- 默认设计尺寸: 1280 x 800.
- 最小建议尺寸: 900 x 680.
- 主内容最大宽度: 760 px.
- 顶部应用栏最大宽度: 920 px.
- 主面板水平居中.
- 主面板使用单列布局.
- 不使用侧边栏.
- 不使用标签页.

```text
+------------------------------------------------------+
| 应用图标和名称                              设置     |
|                                                      |
|        +------------------------------------+        |
|        | 标题和简短说明                     |        |
|        |                                    |        |
|        | APK 文件                           |        |
|        | 自动分析结论                       |        |
|        | 修改前 -> 修改后                   |        |
|        | 调整修复参数, 默认折叠             |        |
|        | 开始自动修复                       |        |
|        +------------------------------------+        |
|                                                      |
|          1 选择 APK -- 2 自动修复 -- 3 输出          |
+------------------------------------------------------+
```

## 6. 顶部应用栏

左侧显示:

- 应用图标.
- `APK Compat Helper`.
- 辅助说明 `旧 APK适配工具`.

右侧只显示设置按钮.

不要增加账户头像, 通知, 帮助中心或主导航.

## 7. 主面板

### 7.1 标题区

固定文案:

```text
第 1 步, 选择 APK
修复旧 APK, 让它能在新设备上安装
文件只在本机处理. 原 APK不会被覆盖.
```

标题直接说明用户可以获得什么, 不使用营销口号.

### 7.2 APK 文件区

未选择文件时显示拖放区域:

```text
拖入 APK, 或点击选择
支持单个 .apk 文件
```

选择文件后显示一行文件信息:

- APK图标.
- 文件名.
- 文件大小.
- 原始 targetSDK.
- APK类型.
- `已解析`状态.
- 移除按钮.

文件名过长时使用省略号, 不允许撑破布局.

### 7.3 自动分析结论

使用浅绿色提示区域:

```text
可以尝试自动修复
将调整安装兼容设置, 补充明确的组件声明, 然后重新签名.
```

这里只显示结论, 不展示完整风险列表.

存在实验性风险时可以增加一句简短说明, 但不得超过两行:

```text
检测到权限风险, 但相关功能可能不会被使用. 该风险不会阻止修复.
```

## 8. 修改前后对比

使用一条紧凑的对比区域:

```text
修改前                       修改后
targetSDK 19       ->         targetSDK 24
Android 4.4                   Android 7.0
```

要求:

- 左右信息保持同级.
- 修改后的 targetSDK使用绿色强调.
- targetSDK变化时实时更新修改后数据.
- 首版只展示影响最大的 targetSDK对比.
- 不在默认界面展示所有 Manifest差异.

## 9. 高级参数

默认折叠, 标题为:

```text
调整修复参数    将修改 2 项, 一般不需要调整
```

展开后只显示:

### 9.1 目标 targetSDK

- 使用原生下拉选择器.
- 同时显示 API Level 和 Android版本.
- 默认选择目标设备允许安装的最低推荐版本.

示例:

```text
API 24 · Android 7.0
```

### 9.2 补充组件导出声明

- 使用开关.
- 默认开启.
- 辅助说明: `仅修改语义明确的主入口`.

### 9.3 允许明文 HTTP

- 使用开关.
- 默认关闭.
- 辅助说明使用警告色: `存在安全风险, 默认关闭`.

### 9.4 权限风险提示

使用浅黄色的小型提示区:

```text
检测到存储权限, 但主题功能可能不会使用. 该风险不会阻止修复.
```

不要在高级参数中显示原始 Manifest XML或 DEX调用详情.

## 10. 主操作

页面只保留一个高强调主按钮:

```text
开始自动修复 ->
```

按钮要求:

- 占据主面板完整宽度.
- 高度至少 54 px.
- 使用橙色背景和白色文字.
- 点击时提供轻微按压反馈.
- 修复执行期间禁用按钮并显示当前阶段.

按钮下方显示:

```text
预计需要 20 到 60 秒 · 将生成新的已签名 APK副本
```

## 11. 底部流程提示

在主面板下方显示轻量流程:

```text
1 选择 APK -- 2 自动修复 -- 3 获取新 APK
```

当前阶段使用绿色, 其他阶段使用灰色.

该流程只用于提供方向感, 不允许成为可点击导航.

## 12. 任务执行状态

点击开始修复后, 保持在同一个主面板内, 使用阶段列表替换表单内容:

```text
正在修复 APK

[完成] 解包 APK
[完成] 应用修改
[进行] 重新构建
[等待] 对齐和签名
[等待] 最终验证

取消
```

要求:

- 显示明确阶段, 不只使用无限旋转动画.
- 用户可以取消.
- 取消后保留日志, 不输出中间 APK.
- 不默认展示原始命令日志.

## 13. 完成状态

成功后在同一主面板中显示:

```text
新 APK已生成
构建, 对齐和签名验证均已通过.

[打开输出目录]
[处理另一个 APK]
```

同时显示:

- 输出文件名.
- 输出目录.
- targetSDK修改前后值.
- 新签名提示.

设备验证作为次要操作, 不放入首要流程.

## 14. 错误状态

错误信息必须包含原因和下一步:

```text
APK重建失败
该 APK的资源表无法被当前版本的 apktool重建.

原 APK未被修改.
[查看日志] [更换 APK]
```

不要只显示错误码, 堆栈或外部工具原始输出.

## 15. 动效

- 只使用 150 到 220 ms 的淡入和状态切换.
- 高级参数使用简单展开和收起.
- 按钮使用轻微位移或缩放反馈.
- 不使用持续浮动, 粒子, 视差或大面积转场.
- 遵循 `prefers-reduced-motion`.

## 16. 可访问性

- 所有按钮提供可访问名称.
- 支持完整键盘操作.
- 焦点状态使用清晰轮廓.
- 不只依赖颜色表达成功和警告.
- 主按钮高度至少 48 px.
- 设置和移除按钮点击区域至少 40 x 40 px.
- 文件选择必须支持点击, 不只支持拖放.

## 17. 禁止增加的内容

- 侧边栏.
- 仪表盘.
- 多标签导航.
- 用户账户和登录.
- 历史任务数据库.
- 批量 APK队列.
- 大面积兼容性时间线.
- 默认展开的完整风险列表.
- 原始 Manifest和 DEX详情.
- 多个同级主按钮.
- 营销文案, 指标卡片或装饰图表.

- [ ] targetSDK选择变化后对比实时更新.
- [ ] 原文件不会被覆盖的提示可见.
